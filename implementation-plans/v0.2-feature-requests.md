## 1. Plus Button On Category To Add To That Category

**Scope:** Medium
**Why it is not trivial:** Section headers are currently presentational only. Add-item state is global, and the detail sheet supports category preselection but nothing in the UI invokes it from a specific category.

### Current state

- Category headers render in `SectionList.renderSectionHeader` inside [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- New item creation supports an explicit `category` only through the detailed editor (`editCategory`)

### Proposed implementation

- Add a small icon button to each non-completed section header
- Tapping the button opens the existing item detail sheet in "new item" mode
- Pre-fill `editCategory` from the section’s category
- Optionally pre-focus the name input so this still feels fast

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- [components/ui/IconButton.tsx](/Users/zacharymarion/Documents/code/cartful/components/ui/IconButton.tsx) if header hit area or labeling needs adjustment

### Risks / decisions

- "Completed" should not get an add button
- Hidden-category items rolled into "Other" will still add to explicit `"other"` unless product wants a chooser instead

### Testing

- Add from Produce header creates an unchecked item in Produce
- Add from Other header respects category `"other"`
- Header action does not interfere with scrolling or sticky behavior

## 2. Show Full Edit Screen When Tapping The Bottom Input

**Scope:** Small to Medium
**Why it is not trivial:** The current bottom bar is optimized for quick-add via `TextInput.onSubmitEditing`, and the detailed flow is secondary via a pencil icon.

### Current state

- Bottom composer contains a live `TextInput` and submit button
- `handleAddItemDetailed()` already opens the bottom sheet and seeds parsed name/quantity/unit

### Proposed implementation

- Change bottom input tap behavior from inline typing to "open detailed add sheet"
- Replace the inline `TextInput` with a pressable faux input, or keep the input read-only and intercept focus
- Preserve the existing quick parsing logic by passing any draft text into the full editor if product still wants keyboard entry first

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)

### Risks / decisions

- This is a workflow change, not just a visual tweak. It removes the fastest add path unless a secondary quick-add affordance remains.
- The current arrow-up submit button becomes redundant if tapping always opens the sheet.

### Testing

- Tapping the bottom composer opens the full add sheet every time
- New item sheet still supports parsed quantity/unit defaults
- Keyboard and `KeyboardStickyView` behavior still feel correct on iOS

## 3. Ability To Reorder Categories Per List

**Scope:** Large
**Why it is not trivial:** Category order is currently global per user via `userPreferences.categoryOrder`; list documents do not store category ordering.

### Current state

- Global category ordering is exposed via `useCategories()`
- `ListDetailScreen` builds sections from `visibleCategories`
- `GroceryList` has no field for per-list category order

### Proposed implementation

- Add `categoryOrder?: GroceryCategory[]` to list documents, separate from user preferences
- Update list fetching/types so each list can override the default category order
- Teach `useCategories()` to accept an optional per-list override or add a list-specific category hook
- Add a per-list reorder UI, likely in the list detail action menu or a dedicated sheet

### Files likely touched

- [types/index.ts](/Users/zacharymarion/Documents/code/cartful/types/index.ts)
- [hooks/useLists.ts](/Users/zacharymarion/Documents/code/cartful/hooks/useLists.ts)
- [hooks/useCategories.ts](/Users/zacharymarion/Documents/code/cartful/hooks/useCategories.ts)
- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- [app/(app)/settings.tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/settings.tsx>) if global ordering remains there too
- `firestore.rules` if list document shape restrictions are explicit

### Risks / decisions

- Need a precedence rule:
  - Recommended: per-list order if present, otherwise user preference order, otherwise default constant order
- Shared lists complicate authorship:
  - Recommended: category order belongs to the list and affects all members
- Existing hidden category preference is still global; decide whether visibility should also become per-list

### Testing

- Reordering categories on one list does not affect another list
- Shared list members see the same category order
- Lists without `categoryOrder` preserve current behavior

## 4. Alphabetically Sort Within Category

**Scope:** Medium
**Why it is not trivial:** The app already has a global `sortOrder` preference, but `ListDetailScreen` currently sorts unchecked items by manual order and creation timestamp inside sections.

### Current state

- `usePreferences()` exposes `sortOrder`
- `ListDetailScreen.sortItems()` ignores `sortOrder` and always prefers `order`, then `createdAt`

### Proposed implementation

- Respect `preferences.sortOrder` inside section sorting
- For `"alphabetical"`, sort by normalized item name within each category, while keeping checked items grouped separately
- Preserve a deterministic fallback when names match

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- [hooks/usePreferences.ts](/Users/zacharymarion/Documents/code/cartful/hooks/usePreferences.ts) only if additional helpers are needed

### Risks / decisions

- If reorder mode is enabled while alphabetical sorting is active, manual drag is conceptually conflicting
  - Recommended: disable reorder mode unless sort order is `"manual"`
- Decide whether completed items also sort alphabetically or retain completion chronology

### Testing

- Alphabetical mode sorts items A-Z inside each category
- Manual mode preserves drag order
- Category grouping still works when names differ only by case

## 6. Autocomplete While Typing Item

**Scope:** Medium
**Why it is not trivial:** History-backed suggestions already exist in `useItemHistory()`, but the list detail screen does not render suggestion UI.

### Current state

- `getSuggestions(searchText)` returns recent matching items
- Suggestions include `name`, `quantity`, `unit`, and `category`

### Proposed implementation

- Render a suggestion list beneath the add composer or inside the full add sheet
- Selecting a suggestion should populate at least name/quantity/unit/category and optionally save immediately
- Use the existing history store first; no new backend needed for V0.2

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- [hooks/useItemHistory.ts](/Users/zacharymarion/Documents/code/cartful/hooks/useItemHistory.ts) if ranking or shape needs adjustment

### Risks / decisions

- This feature interacts directly with request #2:
  - If the bottom composer stops being a text input, autocomplete should move into the full add sheet
- History fetch is one-shot today, not live; that is fine for this use case
- Need to define whether tapping a suggestion adds immediately or opens editable details
  - Recommended: populate the editor, not immediate save

### Testing

- Typing "mi" suggests "Milk" if it exists in history
- Selecting a suggestion preserves stored unit/category
- No suggestions shown when history is empty

## 7. Drag To Reorder Items Between Categories

**Scope:** Large
**Why it is not trivial:** The existing drag implementation only reorders within a section. Persistence is a flat `order` integer and does not update item category during drag.

### Current state

- Reorder mode wraps rows in a vertical pan gesture
- `handleReorder()` accepts `sectionTitle`, `itemId`, and `offset`
- Reordering only mutates the current section array

### Proposed implementation

- Replace the current per-row offset model with a list-aware drag model that understands cross-section targets
- On drop across sections:
  - update item `category`
  - recompute global unchecked ordering
  - persist both category and order
- Keep completed items excluded from drag targets

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- [hooks/useItems.ts](/Users/zacharymarion/Documents/code/cartful/hooks/useItems.ts)

### Risks / decisions

- Current fixed-height drag math is unlikely to scale to cross-section drop zones reliably
- May require introducing a proper drag-and-drop library or a much more complex custom gesture system
- Hidden categories merging into "Other" creates ambiguity on drop destination
  - Recommended: only allow drops into visible concrete categories plus explicit Other

### Testing

- Dragging Produce item into Dairy changes its category and persists after refresh
- Reordering within the same category still works
- Completed items cannot be dragged into active sections

## 8. Undo Button / Toast System

**Scope:** Medium to Large
**Why it is not trivial:** The app currently uses `Alert` for confirmations and haptics for feedback, but there is no transient global feedback system.

### Current state

- Success states are mostly silent or haptic-only
- Destructive actions often require blocking confirmation dialogs

### Proposed implementation

- Add a lightweight app-level toast provider with a queue and optional action button
- Support at least:
  - success/info/error message
  - optional "Undo" action
  - auto-dismiss timing
- Start by wiring it to item deletion and possibly item check/uncheck

### Files likely touched

- [app/\_layout.tsx](/Users/zacharymarion/Documents/code/cartful/app/_layout.tsx)
- New provider/component under `components/ui/` or `lib/`
- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)
- Potentially other screens once the system exists

### Risks / decisions

- Undo requires reversible mutations, not just visual feedback
- Firestore operations are immediate; undo means either delaying destructive commits briefly or storing enough state to recreate deleted entities
  - Recommended V0.2: support undo for delete only, by re-creating the removed item within a short window
- Toast placement must coexist with bottom sheets and sticky keyboard UI

### Testing

- Deleting an item shows a toast with Undo
- Undo restores the item content and category correctly
- Multiple quick actions do not stack in a broken way

## 9. Only Tapping On The Check Should Check Off The Item; Tapping On Name Allows You To Edit Name Inline

**Scope:** Medium
**Why it is not trivial:** The current row makes the entire content area toggle completion, and edit is handled through a separate pencil icon plus a full-sheet editor.

### Current state

- Checkbox tap toggles completion
- Row body tap also toggles completion
- Pencil icon opens full edit sheet

### Proposed implementation

- Remove completion behavior from the row body
- Make row body tap enter inline name edit mode for that item
- Keep quantity/category/note edits in the full-sheet editor, or allow body tap to edit name only and preserve pencil for advanced edits

### Files likely touched

- [app/(app)/list/[id].tsx](</Users/zacharymarion/Documents/code/cartful/app/(app)/list/[id].tsx>)

### Risks / decisions

- Inline editing inside `SectionList` rows adds focus and keyboard-management complexity
- Reorder mode and inline edit mode should be mutually exclusive
- Need a save model:
  - Recommended: auto-save on blur / submit, cancel on outside tap if unchanged
- If the pencil icon remains, define the distinction clearly:
  - Recommended: name tap = quick rename, pencil = full item editor

### Testing

- Tapping item name never toggles completion
- Checkbox still toggles reliably
- Inline rename persists on submit and survives list refresh

## Overall Recommendation

For V0.2, ship requests 1, 2, 4, 5, 6, 8, and 9 first. They are coherent UX improvements and mostly build on existing structures.

Requests 3 and 7 should be treated as structural work. Both touch persistence and ordering semantics, and both will go faster if designed together because category order, item order, and drag/drop destination logic are tightly coupled.
